## 🔐 Enable Leaked Password Protection

### What It Does

**Leaked Password Protection** prevents users from using passwords that have been compromised in data breaches by checking them against the HaveIBeenPwned.org database (over 12 billion compromised passwords).

### Why It's Important

- **Prevents Account Takeover**: Users often reuse passwords across multiple sites
- **Compliance**: Many security standards require checking for leaked passwords
- **Zero Trust**: Assume passwords from breaches will be used in credential stuffing attacks
- **User Protection**: Proactively protects users who may not know their password was compromised

### How to Enable

1. **Go to Supabase Dashboard**:
   - URL: https://supabase.com/dashboard/project/klnspdwlybpwkznzezzd/auth/policies
   - Or navigate manually: Dashboard → Your Project → Authentication

2. **Navigate to Settings**:
   - Click **"Authentication"** in the left sidebar
   - Click **"Settings"** tab

3. **Find Password Settings**:
   - Scroll down to the **"Password Settings"** section
   - Look for **"Leaked Password Protection"** option

4. **Enable the Feature**:
   - Toggle the switch to **ON** (it should turn green/blue)
   - The setting is saved automatically

5. **Verify**:
   - The toggle should remain ON
   - Try creating a test user with a known leaked password (e.g., "password123")
   - The system should reject it

## 📝 Notes

- **No Code Changes Required**: This is a project-level setting
- **Applies to All Users**: Both new signups and password resets
- **External API**: Supabase uses HaveIBeenPwned.org's k-Anonymity API (privacy-preserving)
- **No Performance Impact**: Password checking is done during signup/reset only
- **Cannot Be Disabled via SQL**: Must be configured through dashboard

## 🆘 Troubleshooting

### Issue: Can't Find the Setting

**Solution**: Make sure you're in the correct project (klnspdwlybpwkznzezzd) and have admin access.

### Issue: Toggle Keeps Turning Off

**Solution**: Check if you have sufficient permissions. You may need project owner or admin role.

### Issue: Want to Test Without Rejecting Users

**Solution**: You can't disable it per-user, but you can use a strong unique password for testing (e.g., generated by password manager).
